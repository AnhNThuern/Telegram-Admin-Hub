# PRD — Hệ thống Admin quản trị bán hàng qua Telegram Bot + SePay

## 1. Document Control

- **Project Name:** Telegram Shop Admin
- **Document Type:** Product Requirements Document (PRD)
- **Version:** 1.0
- **Status:** Draft for implementation
- **Primary Audience:** Gemini Pro 3 / AI coding agent / engineering team
- **Language:** Vietnamese
- **Source Constraint:** Chỉ được dựa trên thông tin và UI reference đã cung cấp. Không tự ý thêm feature ngoài scope.

---

## 2. Product Summary

Xây dựng một hệ thống quản trị bán hàng tự động qua **Telegram Bot**, bao gồm:

1. **Website Admin** để quản lý vận hành
2. **Telegram Bot** để khách hàng tương tác và mua hàng
3. **SePay** là phương thức thanh toán chính
4. **Quản lý sản phẩm số** với tồn kho dạng từng dòng stock
5. **Tự động giao hàng** sau khi thanh toán thành công

Sản phẩm cần tái tạo theo mô hình quản trị đã được tham chiếu từ UI mẫu:
- Dashboard
- Sản phẩm
- Đơn hàng
- Giao dịch
- Khách hàng
- Khuyến mãi
- Cấu hình Bot
- Nhật ký Bot
- Thanh toán / Cài đặt

---

## 3. Product Goal

### 3.1 Business Goal
- Cho phép chủ shop bán hàng tự động qua Telegram
- Giảm thao tác thủ công trong quản lý đơn và thanh toán
- Tập trung toàn bộ vận hành trên một trang admin

### 3.2 User Goal
- Admin dễ dàng thêm sản phẩm, quản lý stock, theo dõi đơn hàng
- Khách hàng có thể xem sản phẩm, đặt hàng, thanh toán và nhận hàng qua Telegram bot
- Dòng thanh toán phải rõ ràng, đơn giản, tự động hóa với SePay

---

## 4. Scope

## 4.1 In Scope (MVP)
Hệ thống phải có các module sau:

1. Authentication cho admin
2. Dashboard tổng quan
3. Quản lý sản phẩm
4. Quản lý danh mục
5. Quản lý stock sản phẩm số
6. Quản lý đơn hàng
7. Quản lý giao dịch
8. Quản lý khách hàng Telegram
9. Cấu hình Telegram Bot
10. Nhật ký Bot
11. Tích hợp SePay
12. Telegram Bot flow cơ bản
13. Tự động giao hàng cho sản phẩm số
14. Khuyến mãi cơ bản ở mức cấu hình dữ liệu

## 4.2 Out of Scope
Không triển khai trong bản đầu:
- Multi-tenant / multi-shop
- Affiliate / hoa hồng phức tạp
- Hệ thống gói dịch vụ
- Mobile app
- Phân quyền nhiều vai trò phức tạp
- Rule engine khuyến mãi nâng cao
- CMS / landing page
- Các tính năng ngoài UI và mô tả đã cung cấp

---

## 5. Key Assumptions

1. Hệ thống có **1 admin chính**
2. Telegram là kênh bán hàng chính
3. SePay là phương thức thanh toán chính
4. Sản phẩm trọng tâm là **sản phẩm số** có stock dạng line-by-line
5. Giao hàng thực hiện trực tiếp qua Telegram
6. Web admin là giao diện vận hành, không phải website storefront cho khách

---

## 6. User Roles

## 6.1 Admin
Admin có thể:
- Đăng nhập admin panel
- Xem dashboard
- Quản lý sản phẩm, danh mục, stock
- Xem đơn hàng, giao dịch, khách hàng
- Cấu hình bot Telegram
- Cấu hình thanh toán SePay
- Xem log bot
- Tạo khuyến mãi cơ bản

## 6.2 Customer
Customer sử dụng Telegram bot để:
- Bắt đầu bot qua `/start`
- Xem sản phẩm
- Chọn số lượng
- Tạo đơn hàng
- Nhận hướng dẫn thanh toán
- Thanh toán
- Nhận hàng

---

## 7. Product Flow Overview

### 7.1 Admin Flow
1. Admin đăng nhập
2. Admin cấu hình Telegram bot token
3. Admin cấu hình SePay
4. Admin tạo danh mục
5. Admin tạo sản phẩm
6. Admin nhập stock
7. Admin theo dõi đơn hàng, giao dịch, khách hàng, log

### 7.2 Customer Flow
1. Customer vào bot và gửi `/start`
2. Bot tạo hoặc cập nhật hồ sơ customer
3. Bot hiển thị menu / sản phẩm
4. Customer chọn sản phẩm và số lượng
5. Hệ thống tạo đơn hàng `pending`
6. Hệ thống tạo payment request qua SePay
7. Customer thanh toán
8. Hệ thống nhận webhook
9. Đơn chuyển `paid`
10. Hệ thống cấp hàng tự động
11. Đơn chuyển `delivered`
12. Bot gửi hàng cho customer

- `content`
- `status`

- `available`
- `reserved`
- `delivered`

## Rules
- Mỗi dòng stock chỉ được giao 1 lần

## Acceptance Criteria

# FR-11 SePay Payment Integration

## Objective
Sử dụng SePay làm phương thức thanh toán chính.

## Required Features
- Lưu cấu hình SePay
- Tạo payment request cho order
- Lưu payment reference
- Nhận webhook callback

- Đối soát giao dịch

## Business Rules
- Webhook phải xử lý idempotent
- Không được ghi nhận trùng giao dịch

## Acceptance Criteria
- Tạo được payment request cho order
- Thanh toán thành công làm order cập nhật tự động
- Giao dịch được lưu lại và liên kết đúng

# FR-12 Telegram Bot Commerce Flow

## Objective
Cho phép user mua hàng qua Telegram bot.

## Minimum Bot Flow
1. `/start`
2. Chọn ngôn ngữ (nếu có)
3. Hiển thị menu chính
4. Xem danh sách sản phẩm
5. Xem chi tiết sản phẩm
6. Chọn số lượng
7. Tạo đơn hàng
8. Nhận hướng dẫn thanh toán
9. Sau thanh toán nhận hàng

## Required Bot Capabilities
- Gửi message
- Inline keyboard / button callback
- Ghi log hành động
- Lưu customer context tối thiểu

## Acceptance Criteria
- User thật có thể đi hết flow mua hàng
- Order được tạo từ bot
- Sau payment user nhận được hàng

# FR-13 Auto Delivery

## Objective
Tự động giao hàng cho sản phẩm số sau khi thanh toán thành công.

## Delivery Logic
- Trigger khi order chuyển `paid`
- Lấy stock `available`
- Gắn stock cho order
- Đánh dấu stock là `delivered`
- Gửi nội dung hàng qua Telegram
- Cập nhật order là `delivered`

## Failure Logic
Nếu không còn stock:
- Không giao hàng
- Ghi log lỗi
- Cập nhật trạng thái phù hợp như `needs_manual_action`
- Không retry vô hạn không kiểm soát

## Acceptance Criteria
- Paid order có stock sẽ được giao tự động
- Không giao trùng stock
- Hết stock phải được ghi log rõ ràng

---

## MVP Constraint

## Acceptance Criteria

- Không cần full engine phức tạp ở MVP

## 13. Technical Implementation Constraints for Gemini Pro 3

Gemini Pro 3 phải tuân thủ các nguyên tắc sau khi code:

1. Không thêm package hoặc framework không cần thiết
2. Ưu tiên code dễ maintain, chia module rõ ràng
3. Mọi dữ liệu nhạy cảm dùng environment variables
4. Mọi webhook phải xử lý idempotent
5. Mọi flow giao hàng phải tránh duplicate delivery
6. UI admin phải dark theme, sidebar-based, gần với structure mẫu
7. Khi chưa chắc về một behavior, ưu tiên phương án đơn giản nhất của MVP
8. Không triển khai feature ngoài PRD
9. Sau mỗi phase phải tự mô tả:
   - đã làm gì
   - file nào thay đổi
   - cách test
   - kết quả test
